# Database Schema Design (Cloud SQL PostgreSQL + pgvector)

To support the Automated RAG Evaluator and Diagnoser, the database must store unstructured chunks, dense vector embeddings, and the structured evaluation metrics across various hyperparameter sweeps.

## Core Extension
Ensure the `vector` extension is enabled on the PostgreSQL instance:
```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

---

## Table 1: `document_chunks`
Stores the parsed and chunked corpus (e.g., the HSBC Annual Report) alongside their vector embeddings. The table is designed to hold multiple chunking strategies simultaneously, allowing the `Pipeline Runner` to dynamically filter by `chunking_strategy` during evaluation sweeps.

| Column Name        | Data Type | Description |
| :--- | :--- | :--- |
| `id`               | UUID (PK) | Unique identifier for the chunk. |
| `doc_id`           | VARCHAR   | Source document ID (e.g., 'HSBC_2025_Annual'). |
| `chunking_strategy`| VARCHAR   | The strategy used (e.g., 'fixed_256_overlap_32', 'semantic'). |
| `chunk_index`      | INT       | The sequential order of the chunk in the document. |
| `content`          | TEXT      | The actual extracted text payload. |
| `metadata`         | JSONB     | Additional context (e.g., page number, section header). |
| `embedding`        | VECTOR(768) | The dense vector embedding generated by Vertex AI/Gemini. (Dimension depends on the chosen model). |

**Index Strategy:**
To support efficient dense retrieval (Indexing config: `dense_bge` / `hybrid`), an HNSW or IVFFlat index will be applied to the `embedding` column.
```sql
CREATE INDEX ON document_chunks USING hnsw (embedding vector_cosine_ops);
```

## Table 2: `evaluation_runs`
Stores the metadata for each distinct hyperparameter sweep (a unique combination of configurations from `optimizer_config.json`).

| Column Name         | Data Type | Description |
| :--- | :--- | :--- |
| `run_id`            | UUID (PK) | Unique identifier for the specific experiment run. |
| `timestamp`         | TIMESTAMP | Execution time of the run. |
| `chunking_config`   | VARCHAR   | e.g., 'fixed_512_overlap_64'. |
| `indexing_config`   | VARCHAR   | e.g., 'hybrid_bm25_dense'. |
| `reranking_config`  | VARCHAR   | e.g., 'cross_encoder_miniLM'. |
| `prompting_config`  | VARCHAR   | e.g., 'step_back'. |
| `generation_config` | VARCHAR   | e.g., 'strict_citation_low_temp'. |
| `latency_seconds`   | FLOAT     | Average latency across all queries in this run. |
| `cost_estimate`     | FLOAT     | Estimated API cost (token usage) for this run. |

---

## Table 3: `evaluation_metrics`
Stores the highly granular, per-query metric results generated by the `RAGEvaluator`. This table is the primary input source for the `RAGDiagnoser` engine.

| Column Name       | Data Type | Description |
| :--- | :--- | :--- |
| `id`              | UUID (PK) | Unique row identifier. |
| `run_id`          | UUID (FK) | Links back to `evaluation_runs.run_id`. |
| `dataset_mode`    | VARCHAR   | 'Case1_Direct_Eval' or 'Case2_Proxy_Eval'. |
| `query_id`        | VARCHAR   | The specific query ID (e.g., 'Q001'). |
| `metric_category` | VARCHAR   | 'retrieval', 'generation', or 'end_to_end'. |
| `metric_name`     | VARCHAR   | e.g., 'Recall@5', 'Faithfulness', 'Answer Relevance'. |
| `metric_value`    | FLOAT     | The calculated score (usually normalized between 0.0 and 1.0). |

## 💡 Architecture Note for Interviewers
In a full production environment at Standard Chartered or HSBC, the `evaluation_runs` and `evaluation_metrics` tables would typically be synced to a Data Warehouse (like **Google BigQuery**) using an ELT tool (like Airbyte or Fivetran). This enables the creation of live Dashboard UIs (e.g., Looker or Tableau) for Data Scientists to visually monitor model drift and compare prompt strategies at scale.

## Table 4: `documents` (Corpus Metadata)
Stores high-level metadata for the source documents. This separates document-level properties from the granular chunk data, ensuring a normalized and scalable architecture.

| Column Name     | Data Type | Description |
| :--- | :--- | :--- |
| `doc_id`        | VARCHAR (PK) | Unique document identifier (e.g., 'DOC001', 'HSBC_2025_Annual'). |
| `doc_name`      | VARCHAR   | Human-readable document name (e.g., 'HSBC Annual Report 2025'). |
| `description`   | TEXT      | A brief summary of the document's contents. |
| `created_at`    | TIMESTAMP | Ingestion or publication date. |
| `metadata`      | JSONB     | Extensible document-level metadata (e.g., author, source_url). |

---

## Table 5: `topics`
Stores the distinct topics or domains that documents can be categorized under.

| Column Name   | Data Type | Description |
| :--- | :--- | :--- |
| `topic_id`    | UUID (PK) | Unique identifier for the topic. |
| `topic_name`  | VARCHAR   | The name of the topic (e.g., 'credit_risk', 'banking_regulation'). |

---

## Table 6: `document_topics` (Many-to-Many Mapping)
Resolves the many-to-many relationship between `documents` and `topics`. A single annual report might cover 'credit_risk', 'wealth', and 'retail_banking'.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `doc_id`    | VARCHAR (FK)| Links back to `documents.doc_id`. |
| `topic_id`  | UUID (FK)   | Links back to `topics.topic_id`. |

## 💡 Architecture Note: Metadata Pre-filtering (Hybrid Search)
By introducing the `documents` and `document_topics` tables, this architecture natively supports **Metadata Pre-filtering (or Hybrid Vector Search)**. 
When the RAG Agent processes a user query that implies a specific domain (e.g., *"What are the credit risks..."*), the system can first execute a standard SQL `JOIN` on `document_topics` to isolate the relevant `doc_id`s for 'credit_risk'. The dense vector search (ANN / k-NN) is then executed **strictly within this filtered subset of `document_chunks`**. 
This guarantees that the LLM will not hallucinate answers from irrelevant domains and significantly accelerates the vector search latency over a massive corpus.
